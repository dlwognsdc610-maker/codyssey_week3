## 구현 요약

### 데이터 구조 — `Pattern2D`

| 메서드 | 설명 |
|--------|------|
| `get(r, c)` | 특정 위치 값 읽기 |
| `set(r, c, v)` | 특정 위치 값 쓰기 |
| `rows() / cols()` | 행/열 수 반환 |
| `is_square()` | 정방 행렬 여부 |

### 라벨 정규화

프로그램 내부에서 사용하는 표준 라벨은 `Cross` 와 `X` 두 가지다.

| 원시 입력 | 표준 라벨 |
|-----------|-----------|
| `"+"` | `Cross` |
| `"cross"` | `Cross` |
| `"Cross"` | `Cross` |
| `"x"` | `X` |
| `"X"` | `X` |

`normalize_label()` 함수 하나에서 모든 변환을 처리하므로,  
필터 키와 `expected` 값 모두 동일한 경로를 거쳐 정규화된다.

### MAC 연산

```python
def mac(pattern, filt):
    total = 0.0
    for r in range(pattern.size):
        for c in range(pattern.size):
            total += pattern.get(r, c) * filt.get(r, c)
    return total
```

이중 반복문으로 직접 구현. NumPy 등 벡터화 라이브러리 미사용.

### 동점 처리 정책 (Epsilon 기반)

```python
EPSILON = 1e-9

if abs(score_cross - score_x) < EPSILON:
    verdict = "UNDECIDED"
```

부동소수점 연산에서 두 점수가 수학적으로 같더라도 미세한 오차가 발생할 수 있다.  
단순 `==` 비교 대신 허용 오차(`1e-9`) 범위 내에서 동점으로 간주함으로써  
수치 비교 오류에 의한 오판정을 방지한다.

### 실패 케이스 분석

#### `size_13_2` — FAIL

- **원인 분류**: 데이터/설계 문제
- **상황**: `size_13_2` 패턴은 십자가 선과 X 대각선이 **동시에** 겹쳐있는 복합 패턴이다.  
  결과적으로 Cross 필터 점수 = **25**, X 필터 점수 = **25** 로 완전히 동점이 된다.
- **프로그램 판정**: `UNDECIDED` (epsilon 정책 정상 작동)
- **expected 값**: `+` → 정규화 후 `Cross`
- **FAIL 이유**: 어느 한 쪽을 선택할 수 없는 패턴에 특정 라벨을 기댓값으로 설정했기 때문이다.  
  이는 **로직 오류가 아니라 테스트 데이터 설계상 의도된 경계 케이스**다.
- **수정 방향**: 해당 케이스의 `expected` 를 `"UNDECIDED"` 로 변경하거나,  
  동점 상황에서도 특정 필터를 우선하는 타이-브레이크 정책을 도입할 수 있다.

#### 나머지 7개 케이스가 PASS 인 이유

1. **라벨 정규화**: `"+"` → `Cross`, `"x"` → `X` 변환이 일관되게 적용된다.
2. **Epsilon 정책**: 순수 정수 패턴(0/1)이므로 부동소수점 오차가 발생하지 않고,  
   점수 차이는 항상 `1e-9` 를 크게 초과한다.
3. **필터-패턴 크기 일치**: `size_{N}_{idx}` 키 규칙을 파싱해 자동으로 올바른 필터를 선택한다.

## 성능 표 및 시간 복잡도 분석

### 측정 결과 (10회 평균)

| N×N   | 평균 시간(ms) | 연산 횟수(N²) |
|-------|--------------|--------------|
| 3×3   | ≈ 0.0012     | 9            |
| 5×5   | ≈ 0.0026     | 25           |
| 13×13 | ≈ 0.0137     | 169          |
| 25×25 | ≈ 0.0532     | 625          |

> 실행 환경에 따라 절댓값은 달라지나, **비율은 N² 증가에 수렴**한다.

### O(N²) 근거

MAC 연산의 핵심은 이중 `for` 반복문이다.

```
for r in range(N):       # N 회
    for c in range(N):   # N 회
        total += ...     # O(1) 연산
```

총 곱셈·덧셈 횟수 = **N × N = N²**.  
N 이 2배가 되면 연산 횟수는 4배, 실행 시간도 이에 비례해 증가한다.

실측 비율 검증:

| 비교 쌍 | N² 비율 | 시간 비율(실측) |
|---------|---------|---------------|
| 3→5   | 25/9 ≈ 2.8 | ≈ 2.2 |
| 5→13  | 169/25 ≈ 6.8 | ≈ 5.3 |
| 13→25 | 625/169 ≈ 3.7 | ≈ 3.9 |

시간 비율이 N² 비율에 근접함을 확인할 수 있으며,  
이는 MAC 연산의 시간 복잡도가 **Θ(N²)** 임을 실측으로 뒷받침한다.

실제 AI 모델에서는 수백×수백 크기의 필터를 수천 개 사용하므로,  
단일 이미지 처리에만 수억 번의 MAC 연산이 필요하고,  
이를 병렬로 처리하는 NPU 의 필요성이 여기서 비롯된다.

## Python VM Architecture 기반 상세 분석 (요약 없이 기술)

아래 내용은 이 프로젝트의 MAC 로직 자체(O(N²))와는 다른 축의 분석이다.  
즉, "알고리즘 복잡도"가 아니라 "CPython 인터프리터 실행 방식" 때문에 발생하는 반복 오버헤드를 설명한다.

### 1) CPython 에서 객체가 다뤄지는 방식

CPython 은 대부분의 값을 C 구조체 기반 객체(`PyObject`)로 관리한다.

- 정수, 실수, 문자열, 리스트, 딕셔너리, 사용자 클래스 인스턴스 등은 모두 `PyObject` 계열
- 변수는 객체 본체를 직접 담기보다 객체를 가리키는 참조(포인터)를 가진다
- VM 평가 스택(value stack)에도 객체 본체가 아니라 객체 참조가 push/pop 된다

따라서 실행 중에는 "객체 생성/접근/참조 카운트 갱신"이 계속 발생한다.  
이 특성 때문에 Python 코드의 반복 성능은 단순 산술량뿐 아니라 객체 모델 비용의 영향을 크게 받는다.

### 2 Optimization:

 Python `for` 오버헤드의 핵심 원인을 메모리 대역폭 병목으로 보는 것은 보통 맞지 않는다.  

- 관련이 큰 것: 인터프리터 바이트코드 디스패치, iterator protocol 호출, 객체 타입 분기, 참조 카운트 처리
성능 비용은 "RAM 트래픽"보다 "VM + 객체 접근 절차"에서 더 크게 발생한다.

### 3) `for` 문 1회 반복에서 실제로 일어나는 일

CPython 바이트코드 관점에서 루프는 대략 다음 순서로 진행된다.

1. 반복자 준비
  - `iter(obj)` 개념으로 iterator 객체를 만든다
2. 반복 단계 진입
  - `FOR_ITER` 계열 동작으로 `__next__()` 호출
3. 항목 수신/종료 판단
  - 값이 오면 루프 본문으로 진행
  - 종료 시 `StopIteration` 처리 후 루프 탈출
4. 루프 변수 저장
  - `STORE_FAST` 등으로 지역 변수 슬롯에 바인딩
5. 본문 연산 수행
  - 덧셈/곱셈/인덱싱/함수호출 시 타입 검사, 슬롯 조회, 디스패치 수행
6. 참조 카운트 정리
  - 임시 객체 및 중간 결과에 대해 INCREF/DECREF 경로가 개입
7. 다음 반복으로 점프

중요한 점은, 본문이 매우 단순해도(예: 정수 누적) 위 절차는 반복마다 계속 발생한다는 것이다.

### 4) 왜 "for 문 자체를 줄이면" 성능 근거가 생기는가

Python 에서는 루프 본문 계산량이 작을수록, 고정 오버헤드가 전체 시간에서 차지하는 비중이 커진다.

- 본문이 작은 루프: 오버헤드 비중이 큼 → 반복 수 감소 효과가 큼
- 본문이 큰 루프: 계산 자체가 무거움 → 오버헤드 비중이 상대적으로 작아짐

그래서 Python 성능 최적화에서 자주 나오는 원칙은 아래와 같다.

- "Python 레벨에서 수백만 번 반복"을 피한다
- 가능하면 "한 번에 처리"하는 내장/라이브러리 API로 이동한다
- 결국 반복을 C 레벨 루프로 내려보내는 것이 핵심이다

이 관점에서 "for 문을 줄인다"는 말은 단순 코딩 스타일이 아니라,  
VM 디스패치 횟수와 객체 접근 횟수를 줄이는 정량적 근거를 가진 전략이다.

### 5) 이 현상이 Python 고유 오버헤드인 이유

C/C++/Rust 같은 컴파일 언어는 일반적으로 다음 최적화를 활용한다.

- 루프 전개(unrolling)
- 인라인 최적화
- 레지스터 할당 최적화
- 정적 타입 기반 분기 제거

반면 CPython 은 런타임 동적 타입과 범용 객체 모델을 유지해야 하므로, 같은 단순 반복이라도 매 스텝 처리 절차가 더 길다.  
즉, 여기서 논의하는 오버헤드는 "다른 컴파일 언어와 동일한 조건"의 문제가 아니라, Python 인터프리터 모델에서 특히 두드러지는 비용이다.

### 6) 실무적으로 바로 적용 가능한 판단 기준

다음 조건이면 Python 루프 오버헤드를 우선 의심한다.

- 반복 횟수가 매우 큼(예: 수십만~수백만 이상)
- 루프 본문이 단순(사칙연산/비교/간단 조건)
- 프로파일링에서 특정 루프가 시간 대부분을 점유

개선 우선순위:

1. 내장 함수로 치환 가능한지 확인 (`sum`, `min`, `max`, `any`, `all`)
2. 컴프리헨션/제너레이터로 불필요한 중간 연산 축소
3. 가능하면 C 구현 라이브러리(예: NumPy)로 반복 위임
4. 병목이 계속되면 알고리즘/데이터 구조 자체 재설계

### 7) 이 프로젝트와 연결되는 해석

이 저장소의 MAC 예시는 교육용으로 순수 Python 이중 루프를 사용하므로,  
N 이 커질수록 O(N²) 증가와 함께 Python 루프 오버헤드도 누적된다.

- 알고리즘 차원: 연산 횟수 자체가 N²
- 런타임 차원: 각 연산이 Python 객체/VM 경로를 통과

따라서 "for 문을 줄이면 빨라진다"는 결론은 이 프로젝트에도 유효하다.  
단, 정확한 표현은 "메모리와 완전 무관"이 아니라 "메모리 대역폭보다 VM/객체 오버헤드가 주된 원인"이다.

### 8) 최종 요약 문장 
CPython 에서 `for` 반복의 성능 비용은 주로 인터프리터 바이트코드 디스패치, iterator protocol 호출, 동적 객체 접근/참조 카운트 처리에서 발생한다. 따라서 Python 코드에서 반복을 직접 많이 수행할수록 고정 오버헤드가 누적되며, 반복을 내장 함수나 C 레벨 라이브러리로 위임할 때 성능이 개선된다. 이 현상은 메모리 대역폭 이슈보다 Python VM 실행 모델의 특성에 더 크게 기인한다.
