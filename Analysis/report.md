# CPython 반복 오버헤드 분석 보고서

## 1. 보고서 목적
이 보고서의 목적은 다음 세 가지를 하나의 흐름으로 연결하는 것이다.

1. 실험: 반복문 형태에 따라 실행 시간이 어떻게 달라지는지 확인
2. 분석: 차이가 메모리 문제인지, CPython 실행 모델 문제인지 분해
3. 근거 강화: ASM 수준 자료까지 연결해 원인 후보를 검증


## 2. 실험 질문과 출발 가설
질문:
- 왜 "입력 1개를 도는 for"보다 "입력 2개를 동시에 도는 for"가 느린가?
- 기존 상식으론 메모리 최적화는 보통 더 많은 반복믄을 사용, cache tiling을 사용하지만
  왜 1개의 반복문이 python에서 더 나은 성능을 내는가? 
  python object는 애초에 heap 할당인데 왜 접근 패턴 최적화가 존재하는가? 
  
 
## 3. 기준 성능 측정 결과 (N = 10,000,000)
기존 측정 결과:

| 케이스 | 시간 | 비율 |
|---|---:|---:|
| 이터레이터 1개 (range) | 796ms | 1.00x |
| 이터레이터 2개 (zip + range) | 1140ms | 1.43x |
| 이터레이터 1개 (list) | 1403ms | 1.76x |
| 이터레이터 2개 (zip + list) | 2378ms | 2.99x |

첫 관찰:
- 입력 2개를 동시에 처리하는 루프는 range 기준 약 43% 느려진다.
- list 조합에서는 차이가 더 벌어진다.
- 표면적으로는 메모리 영향이 커 보이지만, 이 단계에서는 확정할 수 없다.

## 4. 코드 레벨 해석: 루프 1회당 추가 작업
기존 분석에서 루프당 차이는 다음처럼 정리된다.

### 4.1 one_iter 경로
- 반복마다 핵심적으로 next 1회
- 루프 변수 저장
- 본문 산술 연산

### 4.2 two_iter(zip) 경로
- 입력 2개에서 값 획득(next) 2회
- 루프 변수 2개 바인딩
- 본문 산술 연산

핵심 재구성 포인트:
- 본질은 "zip이라는 문법 요소"가 아니라 "루프당 처리해야 할 입력 수(1개 vs 2개)"다.
- 입력 수가 2개이면 반복마다 호출/디스패치/객체 처리 경로가 늘어난다.
- N회 반복에서 이 고정 비용이 누적되어 시간 차이가 커진다.

---

## 5. 바이트코드 관점 정리
기존 자료의 바이트코드 요약:

- one_iter: FOR_ITER -> STORE_FAST
- two_iter(입력 2개) 루프: FOR_ITER -> UNPACK_SEQUENCE -> STORE_FAST x2

해석:
- 입력 2개 루프는 반복마다 실행되는 opcode 종류와 수가 더 많다.
- 루프 횟수가 매우 크면 이 차이는 선형 누적으로 커진다.

추가 관찰(no_unpack 관련):
- 튜플 인덱싱(tup[0])이 항상 유리하지 않다.
- UNPACK_SEQUENCE가 특정 조건에서 더 직접적인 경로를 타기 때문에,
  언패킹을 피하는 코드가 오히려 느릴 수 있다.

---

## 6. 원인 후보 재정렬: 메모리 vs 인터프리터
중간 결론은 다음과 같다.

- 현상 설명의 중심은 메모리 대역폭보다 CPython 디스패치 비용에 가깝다.
- 특히 "입력 1개 루프 대비 입력 2개 루프"에서 tp_iternext 호출 횟수가 늘어나는 점이 핵심이다.

입력 2개 루프의 동작 요약:
- 반복마다 값 공급 경로가 2개 필요
- 각 경로마다 iternext/타입 디스패치/refcount 처리 누적
- 값 바인딩 단계도 1개 루프보다 증가

결국 루프 N회에서, "반복당 처리 단위 2배"에 가까운 고정 세금이 누적된다.

## 7. 아키텍처 레이어 관점 분해
기존 perf 문서가 제시한 레이어 분해를 그대로 가져오면:

1. eval loop 레이어
- opcode switch dispatch
- 루프 점프 구간의 상태 체크
- 반복 자체만으로도 고정 비용이 크다.

2. PyObject 디스패치 레이어
- 타입 포인터를 통한 슬롯 디스패치
- INCREF/DECREF 관리
- 동적 객체 모델 유연성의 대가가 반복마다 발생

3. 입력 2개 추가 레이어
- iternext 2회
- 바인딩/언패킹 관련 opcode 추가

요점:
- 단일 원인 1개가 아니라 레이어별 비용이 합산되어 최종 시간 차이를 만든다.

## 8. ASM 수준 검증 자료 (기존 callgrind/objdump 결과)
기존 자료의 핵심은 여기서 가설이 검증된다는 점이다.

### 8.1 총 카운터 비교 (N = 500,000)

| 지표 | one_iter | two_iter(zip) | 비율 |
|---|---:|---:|---:|
| 총 명령어 수 Ir | 629,865,716 | 1,057,627,845 | 1.68x |
| 총 읽기 Dr | 177,831,807 | 301,510,191 | 1.70x |
| 총 쓰기 Dw | 91,460,000 | 156,624,982 | 1.71x |
| LL cache miss | 57,196 | 57,242 | 거의 동일 |

이 표가 주는 결정적 의미:
- 캐시 미스는 거의 같다.
- 따라서 성능 하락의 원인으로 메모리적 구조는 보기 어렵다.
- 반대로, 명령어 수와 객체 관련 작업량 증가는 뚜렷하다.

### 8.2 함수별 증가량(핫패스)
기존 자료 요약:
- _PyEval_EvalFrameDefault: +33M
- PyObject_Malloc: +34M (약 2배)
- PyObject_Free: +48M (약 2배)
- PyLong_FromLong: +19M (약 2배)
- _PyLong_Add: +17M
- zip_next: +21M (신규)

해석:
- 특정 도구(zip) 자체보다 중요한 건 "입력 2개를 동시에 순회"할 때 전체 경로가 커진다는 점이다.
- 객체 생성/해제와 숫자 객체 처리까지 동반 증가한다.
- zip_next 수치는 구현상 관찰 지표일 뿐, 원인 해석의 중심은 "반복당 처리 경로 증가"다.
- 즉 루프 본문 이전의 런타임 고정 비용이 더 많이 발생한다.

### 8.3 ASM 관찰 포인트
기존 자료에서 강조한 포인트:
- range_iternext는 비교적 짧은 경로
- zip_next에서는 간접 호출, refcount 처리, 튜플 생성이 반복
- call *offset(register) 형태의 간접 호출이 루프마다 누적

최종적으로,
- 간접 디스패치 + 객체 수명주기 관리
이 축이 "입력 2개 루프"에서 더 많이 호출되어 오버헤드를 만든다는 결론이 강화된다.

## 9. 최종 결론
이 보고서의 결론은 다음 순서로 이해하는 것이 정확하다.

1. 실측상 입력 2개를 동시에 처리하는 루프가 분명히 느리다.
2. 단순히 메모리 접근 때문이라는 설명은 callgrind 지표와 맞지 않는다.
3. 핵심 병목은 CPython 반복 실행 경로의 고정 비용 누적이다.
4. 특히 입력 1개 루프 대비 tp_iternext 호출 증가와 디스패치/refcount 누적이 큰 비중을 차지한다.
5. 따라서 이 사례의 성능 차이는 메모리 대역폭 문제라기보다,
  CPython의 유연한 동적 객체 모델에서 "for당 처리 단위가 늘어날 때" 지불하는 런타임 비용으로 해석하는 것이 타당하다.
